# bw2ssh

A Python port of [ssh-use](https://code.vexcited.com/vexcited/ssh-use) by [Vexcited](https://vexcited.com), originally written in Rust.

Reads SSH key items from your Bitwarden vault and generates `~/.ssh/config` automatically, using custom fields on each vault item to define which hosts and user each key applies to. Public keys are written to `~/.ssh/` so the SSH client can resolve `IdentityFile` references — private keys remain in the Bitwarden SSH agent and never touch disk.

## Motivation

The [Bitwarden SSH agent](https://bitwarden.com/help/ssh-agent/) tries every key in your vault when connecting to a host. Most SSH servers have a `MaxAuthTries` limit, so if you have more than a handful of keys you'll get locked out before the right one is tried.

The fix is an `~/.ssh/config` that maps each host to its specific key:

```
Host myserver
  IdentitiesOnly yes
  IdentityFile ~/.ssh/bw_abc123.pub
```

Maintaining that file by hand across multiple machines is tedious. bw2ssh generates it automatically from your vault.

## Installation

```bash
# Clone the repo
git clone https://github.com/lzukanovic/bw2ssh.git

# System-wide (requires sudo)
sudo install -m 755 bw2ssh/bw2ssh.py /usr/local/bin/bw2ssh

# Current user only (no sudo) — make sure ~/.local/bin is in your PATH
install -m 755 bw2ssh/bw2ssh.py ~/.local/bin/bw2ssh
```

### Update

```bash
cd bw2ssh
git pull
sudo install -m 755 bw2ssh.py /usr/local/bin/bw2ssh  # or the user-only path
```

You can check what version you currently have with:

```bash
bw2ssh --version
```

### Uninstall

```bash
# System-wide
sudo rm /usr/local/bin/bw2ssh

# Current user only
rm ~/.local/bin/bw2ssh

# Clean up any generated files in ~/.ssh
rm -f ~/.ssh/bw_*.pub ~/.ssh/config
```

## Vault setup

For each SSH key item you want bw2ssh to manage, add the following **custom fields** in Bitwarden:

| Field      | Required | Description                                                                                   |
| ---------- | -------- | --------------------------------------------------------------------------------------------- |
| `HOSTS`    | Yes      | Comma-separated host patterns — maps to the `Host` directive. E.g. `192.168.1.10, rpi`        |
| `HOSTNAME` | No       | The real hostname SSH connects to — maps to `HostName`. Useful when `HOSTS` contains aliases. |
| `USER`     | No       | SSH username — maps to `User`.                                                                |

The item must be of type **SSH Key** in Bitwarden (not a password or note). The public key is read directly from the vault item.

## Usage

```bash
# If you're self-hosting (Vaultwarden etc.), set your server first (one-time)
bw config server https://yourvault.example.com

# One-time login
bw login

# Pull the latest vault state
bw sync

# Unlock and capture the session token
export BW_SESSION=$(bw unlock --raw)

# Run bw2ssh — picks up BW_SESSION automatically
bw2ssh

# Or pass the token explicitly
bw2ssh --token "$BW_SESSION"
```

### Options

```
-t, --token TOKEN   Bitwarden session token. Falls back to $BW_SESSION if not set.
    --dry-run       Preview what would be written without touching any files
-v, --version       Print the current version and exit
```

## What it does

1. Fetches all items from your vault using the `bw` CLI.
2. Filters to items of type SSH Key that have a `HOSTS` custom field.
3. Cleans up any previously generated `bw_*.pub` files and `~/.ssh/config`.
4. Writes each public key to `~/.ssh/bw_<hash>.pub` (mode `600`).
5. Generates `~/.ssh/config` (mode `600`) with one block per key.

The filename hash is derived from SHA-256 of the public key, so filenames are stable across runs as long as the key itself doesn't change.

### Generated config

```
# THIS FILE HAS BEEN GENERATED BY bw2ssh.
# DO NOT EDIT IT!

Include config.d/*

# My Server
Host myserver rpi
  HostName 192.168.1.10
  User dietpi
  IdentitiesOnly yes
  IdentityFile ~/.ssh/bw_2d9c8fd9e861aa68.pub
```

### Persisting your own entries

Place any hand-crafted config files in `~/.ssh/config.d/` — they are included via the `Include` directive at the top and will survive every regeneration.

```bash
mkdir -p ~/.ssh/config.d

cat > ~/.ssh/config.d/personal.conf << 'EOF'
Host my-nas
  HostName 192.168.1.50
  User admin
  IdentityFile ~/.ssh/id_ed25519
EOF
```
